  <action>
Create `tests/test_responsive.js` — automated responsive behavior test suite validating VAL-03. Tests verify that the sidebar collapse breakpoint at 1024px works correctly and touch targets are adequate.

**Test approach:**
- Tests must manipulate the viewport size programmatically. In browser, this is possible via:
  - `window.innerWidth` cannot be directly set — instead, use an iframe or resize the actual browser window
  - **Practical approach for this vanilla JS project:** Use CSS `@media` query simulation via `matchMedia()`. For tests that need actual layout computation, document the manual verification steps AND provide the automated `matchMedia` checks.
  - Use `window.matchMedia('(max-width: 1024px)').matches` to determine expected behavior
  - For computed style checks: query the actual DOM element's computed styles and verify they match the expected responsive behavior

**Test structure:**
- Import test-helpers.js
- Organize into describe blocks: "Desktop Viewport (>1024px)", "Mobile Viewport (≤1024px)", "Breakpoint Boundary (1024px)", "Touch Targets"
- Export `runResponsiveTests()` function

**Desktop tests (>1024px):**
- Verify `document.getElementById('sidebar')` is visible (check: computed `display` is not `none`, `visibility` is not `hidden`, `transform` does not translate off-screen)
- Verify `document.getElementById('backdrop')` does NOT have `.open` class
- Verify sidebar has non-zero width (computed `width` ≈ 280px)

**Mobile tests (≤1024px):**
- Verify sidebar's responsive CSS is applied: check that `getComputedStyle(sidebar).position === 'fixed'` or the sidebar has `transform: translateX(-100%)` by default
- Call `toggleSidebar(true)` → verify sidebar gets `.open` class and backdrop gets `.open` class
- Call `toggleSidebar(false)` → verify both `.open` classes are removed
- Verify backdrop click calls `toggleSidebar(false)` — check the `onclick` attribute on `#backdrop`

**Breakpoint boundary test (1024px):**
- Use `window.matchMedia('(max-width: 1024px)').matches` to detect current state
- Assert that at exactly 1024px, the mobile behavior applies (≤1024 = mobile, >1024 = desktop per RECON.md)

**Touch target tests:**
- Query all `.nav-link` elements and action buttons (`.btn-3d`, `.quick-card`, etc.)
- On a simulated mobile viewport, verify their bounding rect dimensions ≥44x44px (Apple HIG minimum) — OR document that visual verification is needed
- If bounding rect measurement is unreliable without actual viewport resize, mark these tests as "informational" (log dimensions without failing) and note that manual verification is required

**CSS reference (from RECON.md §1.1.2 and responsive.css):**
- Breakpoint: `@media (max-width: 1024px)`
- Sidebar mobile: `position: fixed; inset: 0 auto 0 0; z-index: 50; transform: translateX(-100%); width: 280px`
- Sidebar open: `.sidebar.open { transform: translateX(0); }`
- Backdrop mobile: `.backdrop { display: none; } .backdrop.open { display: block; position: fixed; inset: 0; background: rgba(0,0,0,.55); z-index: 40; }`
- Transition: `transition: transform .25s ease`

**Important:** Do NOT rely on `window.resizeTo()` — it only works on windows opened by `window.open()`. Instead, use `matchMedia` for media-query-dependent tests and document the need for manual viewport verification for computed layout tests.
</action>
